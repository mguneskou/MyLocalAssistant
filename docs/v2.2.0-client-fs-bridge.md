# v2.2.0 design — Client-side filesystem bridge (Option 3)

> Companion to v2.1.7's per-user `WorkRoot`. WorkRoot puts skill files on the
> **server PC** (great for the Admin who runs the server). This bridge puts
> skill files on the **tester's own PC** (great for everyone else) without
> exposing SMB shares or requiring write-back over HTTP uploads.

## Problem

Skills (Excel, future ones) call `System.IO` directly. Today that always means
the server's disk. Testers need files to land in `C:\Users\them\...` on **their**
machine. They also need files they double-click in their own Explorer to be
visible to the assistant.

`WorkRoot` (v2.1.7) only solves it when the user is the same person running
the server.

## High-level shape

A second long-lived connection from each Client to the Server, dedicated to
**reverse RPC**: the server calls methods that execute on the client.

```
Client.exe  <===WebSocket===>  Server
                  |
                  | server -> client request : { id, method:"fs.read", params }
                  | client -> server response : { id, ok|error, result }
                  | client -> server event    : { kind:"fs.changed", path } (later)
```

Skills running server-side call a thin `IClientFs` facade. The facade looks up
the active reverse-RPC channel for that user/conversation and forwards the
call. If no channel is connected, the facade returns a clean error and the
skill can fall back to `WorkRoot` or refuse.

## Why a second channel and not piggybacking on `/api/chat/stream`

- SSE is one-way. We'd have to invent a parallel POST channel for replies and
  correlate by id. That reinvents websockets, badly.
- The chat stream is per-message (opens, streams, closes). The fs channel
  needs to be **persistent for the whole session** so a skill can call it long
  after the user message finished streaming.
- Auth is the same JWT either way; the cost of a second WS is negligible.

## Endpoints

- `GET /api/client/bridge` — upgrade to WebSocket. Authenticated by JWT
  (existing scheme; pass via `Sec-WebSocket-Protocol` because browsers can't
  set Authorization on WS — but our Client is WinForms, so we'll just use the
  `Authorization` header normally).
- Server keeps a `ConcurrentDictionary<Guid userId, ClientBridge>` of active
  bridges. One bridge per Client process. If the user opens two Clients, the
  newer one replaces the older one (older one gets a graceful `bridge.evicted`
  frame and closes).

## Wire format

JSON, one object per WS message. Two roles:

**Server → Client (request):**
```json
{ "t": "req", "id": "01J...", "method": "fs.read", "params": { "path": "C:\\Users\\bob\\report.xlsx", "offset": 0, "length": 65536 } }
```

**Client → Server (response):**
```json
{ "t": "res", "id": "01J...", "ok": true,  "result": { "bytesB64": "..." } }
{ "t": "res", "id": "01J...", "ok": false, "error": { "code": "fs.notFound", "message": "..." } }
```

`id` is a ULID generated by the server. Response correlates by `id`.
Timeout: 30s default (skill can override). On timeout the server-side `Task`
faults with `ClientFsTimeoutException`.

## Methods (v2.2.0)

| method        | params                          | result                                  |
|---------------|---------------------------------|-----------------------------------------|
| `fs.list`     | `path`, `pattern?`, `recursive?`| `entries: [{name,isDir,size,mtime}]`    |
| `fs.read`     | `path`, `offset?`, `length?`    | `bytesB64`, `eof`                       |
| `fs.write`    | `path`, `bytesB64`, `append?`   | `bytesWritten`                          |
| `fs.mkdir`    | `path`                          | `created` (bool)                        |
| `fs.move`     | `from`, `to`, `overwrite?`      | (empty)                                 |
| `fs.delete`   | `path`, `recursive?`            | (empty)                                 |
| `fs.stat`     | `path`                          | `exists`, `isDir`, `size`, `mtime`      |
| `fs.tempPath` | (none)                          | `path` (per-conversation temp dir)      |

Large files: `fs.read`/`fs.write` are chunked at the **caller** (skill). Each
call is one chunk. Default chunk = 256 KiB. Keeps individual frames small and
cancellable.

## Security — non-negotiable, enforced **on the client**

The client picks **one** root folder via `FolderBrowserDialog` ("Folder the
assistant may use on this PC"). It is stored in the user's local config
(`%LOCALAPPDATA%\MyLocalAssistant\Client\bridge.json`) along with a friendly
name. Until the user picks a root, every `fs.*` call returns
`fs.notConfigured` and the server-side skill surfaces a clear error.

Every incoming request path is canonicalized via `Path.GetFullPath`, then
checked: the canonical path **must** start with `<root>\` (case-insensitive
on Windows) and **must not** contain `..` segments before canonicalization.
Symlinks: rejected (`File.GetAttributes` → `ReparsePoint` → deny).

Per-extension write filter (config, default-deny for executables):
`.exe .dll .bat .cmd .ps1 .vbs .js .msi .scr .com` are read-only via the
bridge. Configurable but with a stern "Are you sure?" UI prompt the first
time.

## UI

**Client:**
- New tray menu entry "Assistant file folder…" → folder picker. Shows current
  root if set.
- New status row in main window: `📁 D:\BobScratch  [change]  [disable]`.
- First-run prompt the first time a skill needs filesystem access.

**Admin:**
- Per-agent toggle "May access client filesystem" (default off). When off,
  even if the client has configured a root, the skill won't get bridge
  access for that conversation. (Defense in depth — admins keep control.)

## Server changes

- New `ClientBridgeHub` singleton service. Owns the dictionary of active
  bridges. Exposes `IClientFs GetFor(Guid userId)` returning a proxy.
- `ChatEndpoints` adds the active connection's bridge into the
  `ChatService.Callbacks` so skills can reach it.
- `ChatService.ResolveWorkDirectoryAsync` gains a third path: if the agent
  has `PreferClientFs=true` and a bridge is connected, use
  `client.tempPath()` as `workDirectory` and pass the bridge as an `IClientFs`
  in the skill's `IServiceProvider`.
- Skills: Excel plug-in already takes its work dir from the host. We expose
  `IClientFs` as an optional service. The plug-in's `OpenWorkbook` etc. will
  go through it when present.

## Migration / compatibility

- Server, Admin, Client all gain `2.2.0` versions in lockstep (the wire
  contract is new; mismatched clients just won't have the bridge endpoint).
- Old clients: keep working exactly as today (server-side `WorkRoot` path).
- New `Agents.PreferClientFs` boolean column — handled with the same
  in-place `ALTER TABLE` pattern as `Users.WorkRoot`.

## Out of scope for v2.2.0 (queued for later)

- File watchers / change notifications back to the server.
- Streaming reads with backpressure (just chunk at the skill for now).
- Multiple roots per user.
- Linux/macOS Client (none today).
- mTLS for the bridge (JWT is sufficient because the WS is over the same
  TLS-terminated reverse proxy as the rest of the API).

## Build order

1. `ClientBridgeHub` + WS endpoint, no methods, just connect/disconnect/auth
   + heartbeat.
2. Wire format + request/response correlation + 1 method (`fs.stat`) end to
   end. Smoke test from a unit test that fakes a client.
3. Client: WS connect on login, reconnect on drop, root picker UI.
4. All `fs.*` methods on client side.
5. `IClientFs` facade + `ChatService` integration + `Agents.PreferClientFs`.
6. Excel plug-in: optional `IClientFs` consumer.
7. Ship 2.2.0.
