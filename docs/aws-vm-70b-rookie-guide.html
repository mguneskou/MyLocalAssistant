1) Read This First (70B Reality Check)

Running a local 70B model is heavy. It needs significant memory and often a very expensive GPU instance.

The built-in local 70B entry in this product catalog is approximately 40 GB model file size, with recommended context 8192 and minimum RAM 48 GB. In real usage, budget more than the minimum.

For beginners, there are two valid ways to "use 70B" with this software:

  1. Plan A (Local 70B on your AWS VM): highest control, highest cost, highest setup complexity.
  2. Plan B (Cloud 70B provider from within the app): much cheaper and easier, still accessed from your AWS VM.
If you are not sure, start with Plan B first. You can validate your test flow without paying for top-tier GPU compute immediately.

2) Choose Your 70B Plan

Plan A: Local 70B model in AWS VM

  • Pros: model stays on your VM, no per-token cloud API billing.
  • Cons: very high EC2 cost, quota approvals, GPU setup complexity.
  • Best for: controlled test labs, strict data locality testing, performance profiling.

Plan B: 70B cloud model from inside MyLocalAssistant

  • Pros: easiest startup, lower fixed cost, no massive local model downloads.
  • Cons: prompts/responses leave your VM network to provider API.
  • Best for: functional testing and user acceptance with lower infra overhead.

Hardware guidance table (for local 70B)

LevelSuggestionNotes
Minimum practical GPU with large VRAM, strong RAM headroom May work, but can be unstable with large context or heavy concurrency.
Recommended stable target 80 GB-class GPU memory per workload lane Safer for 70B quantized model plus context/cache overhead.
Budget fallback High-memory CPU instance (slow) Good for correctness checks, not for speed.
If your tester only needs to verify functionality (not throughput), Plan B is usually the fastest path to success.

3) Understand Cost Before Launch

3.1 Cost components

  • EC2 runtime (per hour while instance is running)
  • EBS storage (per month while disk exists)
  • Data transfer (small for basic testing, but monitor anyway)
  • Optional cloud model API usage (if Plan B)

3.2 Set budget guardrails now

  1. AWS Console -> Billing and Cost Management.
  2. Enable billing alerts.
  3. Create monthly budget alerts at 50%, 80%, and 100%.
  4. Add at least two emails for notifications.

3.3 Rookie-safe strategy

Day 1: launch a smaller Windows VM first, install and test app workflow.
Day 2: switch to expensive GPU instance only when app/setup process is proven.
Biggest mistake: leaving expensive GPU VM running overnight/weekends.

4) Create AWS Account Safely (If New)

4.1 Sign up

  1. Go to aws.amazon.com.
  2. Create account with your company email.
  3. Complete phone + payment verification.

4.2 Secure root account immediately

  1. Top-right account menu -> Security credentials.
  2. Enable MFA for root user.
  3. Store backup codes securely.

4.3 Pick one region

Use one region close to your tester location (for example eu-west-2 or us-east-1). Keep everything in this single region.

4.4 Create a key pair for RDP password decryption

  1. EC2 -> Key pairs -> Create key pair.
  2. Name: mla-ec2-key
  3. Type: RSA, format: PEM.
  4. Download and save it safely.
If you lose the PEM file, recovering Windows administrator password is harder.

5) Request GPU Quota (Plan A only)

If you want local 70B on AWS GPU, quota approval may be required before launch.

5.1 Open quota page

  1. AWS Console -> Service Quotas.
  2. Select service: EC2.
  3. Search for instance family quota for your target GPU family (for example P-family or G-family).

5.2 Request increase

  1. Open quota item.
  2. Request a value enough for one test instance.
  3. Submit and wait for approval email.

5.3 Keep fallback path ready

Quota can take time. If blocked, continue with Plan B (cloud 70B model) using a normal VM.

6) Launch EC2 Windows VM

6.1 Open launch wizard

  1. EC2 dashboard -> Launch instance.

RulePortSourceReason
RDP3389Your public IP onlyRemote desktop management.

Do not leave RDP open to the whole internet for ongoing use.

6.4 Launch and wait

  1. Click Launch instance.
  2. Open Instances list.
  3. Wait for 2/2 status checks.

6.5 Allocate Elastic IP

  1. EC2 -> Elastic IPs -> Allocate Elastic IP.
  2. Associate to your instance.

This keeps URL stable after stop/start cycles.

7) Connect with Remote Desktop (RDP)

7.1 Get Windows password

  1. Instances -> select your VM -> Connect -> RDP client tab.
  2. Download RDP file.
  3. Click Get password.
  4. Upload PEM key and decrypt.

7.2 Login

  1. Open .rdp file.
  2. Enter credentials from previous step.
  3. Accept certificate warning.
First login can be slow because Windows runs initial configuration.

8) Install and Validate GPU Driver (Plan A local 70B)

8.1 Install latest Windows updates first

  1. Settings -> Windows Update -> install all updates.
  2. Reboot.

8.2 Install NVIDIA driver for your EC2 GPU family

  1. Open AWS docs search: "AWS EC2 NVIDIA Windows driver".
  2. Open the official AWS driver installation page for your instance family.
  3. Download the recommended driver package.
  4. Run installer and reboot.

8.3 Validate GPU

Open PowerShell and run:

nvidia-smi

You should see GPU model, driver version, and memory stats.

8.4 Set page file for safety

  1. System Properties -> Advanced -> Performance -> Settings.
  2. Advanced tab -> Virtual memory -> Change.
  3. Set a larger fixed page file (for example 64-128 GB depending on disk size).
  4. Reboot.
If driver is missing, local 70B may still run with CPU offload but performance can be very poor.

9) Install MyLocalAssistant

9.1 Download and install

  1. On VM browser, open your GitHub Releases page.
  2. Download latest Setup EXE.
  3. Run installer.

9.2 Launch and open pages

  1. Start MyLocalAssistant.ServerHost.
  2. Use tray menu -> Open Admin.
  3. Use tray menu -> Open Chat.

9.3 First login

  1. Login with bootstrap admin credentials from your internal docs.
  2. Change password immediately.
  3. Create dedicated tester users.

9.4 Confirm server is reachable

On VM PowerShell:

Test-NetConnection -ComputerName localhost -Port 8080

If TcpTestSucceeded is true, app port is listening locally.

10) Load and Test 70B Model

10.1 Local 70B (Plan A)

  1. Admin -> Models tab.
  2. Find local 70B entry (for example Llama 3.3 70B Q4_K_S).
  3. Click Download and wait fully complete.
  4. Click Activate.
  5. Wait for model status to become loaded.

10.2 Cloud 70B (Plan B)

  1. Admin -> Server Settings -> set cloud key (provider you use).
  2. Models tab -> choose 70B cloud entry and Activate.
  3. Confirm model status loaded/configured.

10.3 Quick verification prompts

  1. "Write a 300-word technical summary of a system migration plan."
  2. "Create a risk table with severity and mitigation."
  3. "Generate a Word memo named aws-70b-test.docx."

10.4 If local 70B fails to load

  • Check logs for memory/backend error details.
  • Verify driver and nvidia-smi.
  • Reduce workload (single user, lower concurrent activity).
  • Temporarily switch to cloud 70B for functional testing.

11) Security Hardening Checklist

  • AWS root account MFA enabled.
  • RDP port 3389 restricted to trusted admin IP(s) only.
  • Port 8080 restricted to tester IPs whenever possible.
  • Default app passwords changed.
  • Only required users are admin/global-admin.
  • Budget alerts configured and tested.
  • Elastic IP documented.
  • VM stop/start ownership clearly assigned.
Never share RDP admin credentials in plain text email or chat.

12) Daily Operation Checklist

Start of day

  • Start EC2 instance.
  • RDP login succeeds.
  • Open Admin and Chat pages.
  • Model status is loaded.
  • Test one prompt response.

End of day

  • Export any required logs or screenshots.
  • Stop EC2 instance if not actively testing.
  • Confirm instance is in Stopped state.
Stopped instance still incurs EBS storage cost. Running instance incurs full compute cost.

13) Troubleshooting Guide

Problem: Cannot RDP into VM

  • Check instance state is Running.
  • Check status checks are 2/2.
  • Verify security group allows your current public IP on 3389.
  • If your IP changed, update SG rule.

Problem: Tester cannot open app URL

  • Ensure app is running on VM.
  • Check security group inbound for 8080 from tester IP.
  • From VM browser test http://localhost:8080/.
  • Use Elastic IP in URL, not old temporary public IP.

Problem: 70B activation fails

  • Open app logs and read exact error.
  • Check available RAM and GPU memory.
  • Check GPU driver using nvidia-smi.
  • Try cloud 70B temporarily to continue tester cycle.

Problem: Costs are too high

  • Verify instance was not left running.
  • Check if more than one VM is active.
  • Consider switching from local 70B to cloud 70B for routine validation.
  • Tighten test windows (scheduled test sessions only).

14) How to Stop Billing Safely

Pause testing

  1. EC2 -> Instances -> select VM.
  2. Instance state -> Stop instance.
  3. Confirm state changes to Stopped.

Permanent cleanup

  1. Back up anything needed first.
  2. Terminate instance only when done permanently.
  3. Delete unused EBS volumes and Elastic IP if no longer required.
Terminate is destructive. Stop is safer when uncertain.
